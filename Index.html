Lotka–Volterra in Python

Building Euler's method and 4th-order Runge–Kutta from scratch — with the math on one side and the code on the other.

1

The model

▾

The Lotka–Volterra system describes how two populations — one a predator, one its prey — rise and fall over time. Prey reproduce on their own, but predators eat them. Predators need prey to survive, so when prey become scarce, predators starve. Prey recover, predators come back, and the cycle repeats. The model captures this with two coupled ordinary differential equations and four parameters.

SymbolValueMeaning
α1.0Prey growth rate
β0.4Predation rate
δ0.2Predator reproduction efficiency
γ0.5Predator death rate
Imports and plot defaults
import numpy as np
import matplotlib.pyplot as plt

plt.rcParams['figure.figsize'] = (10, 6)
plt.rcParams['font.size'] = 11

NumPy gives us MATLAB-style arrays for numerical work. By convention it is imported as np. Matplotlib is the plotting library; pyplot is the interface that mirrors MATLAB's plotting commands and is conventionally imported as plt.

The two rcParams lines set defaults so every figure we make later comes out at the same size with the same font. Optional, but it keeps the output consistent.

Model parameters
alpha = 1.0   # prey growth rate
beta  = 0.4   # predation rate
delta = 0.2   # predator reproduction efficiency
gamma = 0.5   # predator death rate

The four parameters that govern the system. We define them at the top of the script as plain variables — no class, no dictionary, no configuration file. For a notebook-style script this is the right level of complexity.

These values are arbitrary but well-behaved: they produce clean oscillations that are easy to see and easy to integrate. Real ecological data would use very different numbers, but the structure of the equations stays the same.

The prey equation
def dxdt(x, y):
    """Prey equation: dx/dt = alpha*x - beta*x*y"""
    return alpha * x - beta * x * y
$$\frac{dx}{dt} = \alpha x - \beta x y$$

The rate of change of the prey population. The first term, $\alpha x$, is exponential growth — prey reproduce on their own, and the more there are, the faster more are born. The second term, $-\beta x y$, is predation: it depends on the product of prey and predator counts because both have to be present for an encounter to happen, and the minus sign says encounters reduce the prey population.

Note: the function uses alpha and beta without taking them as arguments. In Python, a function can read variables defined in the surrounding scope. This is different from C++ (where it would be a scope error) and from MATLAB (where local functions can't see the caller's workspace). For a notebook script it is the idiomatic choice and keeps the function signature clean.

The predator equation
def dydt(x, y):
    """Predator equation: dy/dt = delta*x*y - gamma*y"""
    return delta * x * y - gamma * y
$$\frac{dy}{dt} = \delta x y - \gamma y$$

The rate of change of the predator population. The first term, $\delta x y$, is the predator birth term — predators only reproduce when they have food, so the rate is tied to the same $x y$ interaction as the prey loss term, scaled by how efficiently prey are converted into new predators. The second term, $-\gamma y$, is the death rate: predators die off on their own when there is no food.

Both dxdt and dydt take the full state $(x, y)$ as input even though some terms only use one of the two variables. This keeps the function signatures uniform and makes the solver code in later sections cleaner.

A note on solver design

Notice that we are writing the right-hand sides of the equations as their own functions, separate from any solver. This is not just stylistic — it is the design that lets us reuse the same Euler or RK4 code for any other system of two ODEs. The solver does not know or care what dxdt actually computes; it just calls it with the current state and uses the returned slope.

If you wanted to integrate, say, the SIR epidemic model instead, you would replace dxdt and dydt with the SIR right-hand sides and leave the solver code untouched. This separation between what the equations are and how to step them forward in time is one of the most important habits in scientific computing.

2

Equilibrium sanity check

▾

An equilibrium is a state where nothing changes — both rates of change are exactly zero. Setting $dx/dt = 0$ and $dy/dt = 0$ and solving the two equations gives a non-trivial fixed point at $(x^*, y^*) = (\gamma/\delta,\ \alpha/\beta)$. With our chosen parameters, that works out to $(2.5, 2.5)$. If we plug those values into dxdt and dydt and the results are exactly zero, we know the equations were typed correctly. A two-minute habit that catches typos before they become confusing plots.

Compute the equilibrium
x_eq = gamma / delta  # 2.5
y_eq = alpha / beta   # 2.5
Set both rates to zero and solve: $$\alpha x - \beta x y = 0 \implies y^* = \alpha/\beta$$ $$\delta x y - \gamma y = 0 \implies x^* = \gamma/\delta$$

The math: factor each equation. The prey equation factors as $x(\alpha - \beta y) = 0$, which is zero when $y = \alpha/\beta$. The predator equation factors as $y(\delta x - \gamma) = 0$, which is zero when $x = \gamma/\delta$. The trivial solution $(0, 0)$ is also a fixed point but a boring one — we want the coexistence equilibrium where both populations are nonzero.

Evaluate the derivatives at the equilibrium
print(f"Equilibrium: ({x_eq}, {y_eq})")
print(f"dx/dt at eq: {dxdt(x_eq, y_eq)}")
print(f"dy/dt at eq: {dydt(x_eq, y_eq)}")
$$f_x(x^*, y^*) = 0$$ $$f_y(x^*, y^*) = 0$$

If both prints come out as 0.0, the right-hand sides are typed correctly. If either is nonzero, there is a sign or factor error somewhere — fix it before going any further.

About f-strings. The f prefix before the string lets you embed variables in braces: f"value: {x}" evaluates x and substitutes it into the string. This is Python's equivalent of MATLAB's fprintf or C++'s printf, but cleaner because the variables go inline rather than as separate arguments.

3

Euler's method

▾

Here is the core idea, in one sentence: if you know your current position and your current velocity, a short time later you will be approximately at position-plus-velocity-times-time. That is Euler's method. Applied to our system, each step uses the current state to compute the slopes $dx/dt$ and $dy/dt$, then nudges both populations forward by dt times those slopes. We repeat the nudge until we reach the final time.

Function signature and step count
def euler_method(x0, y0, dt, t_final):
    n_steps = int(t_final / dt)
$$N = \frac{T}{\Delta t}$$

The four arguments are the inputs the math needs: a starting state $(x_0, y_0)$, a step size $\Delta t$, and a final time $T$. The number of steps is the total time divided by the step size — int() truncates to a whole number because you can't take half a step.

Pre-allocate the result arrays
    t = np.zeros(n_steps + 1)
    x = np.zeros(n_steps + 1)
    y = np.zeros(n_steps + 1)
$$t_0,\ t_1,\ t_2,\ \ldots,\ t_N \quad \rightarrow\ N+1\ \text{points}$$

We need an array slot for every time point we'll visit, including the starting one. Why $N+1$? Taking $N$ steps produces $N+1$ points — count the fence posts, not the fence sections.

Why pre-allocate? Faster than appending. Same idea as avoiding vector::push_back in a tight C++ loop when the final size is known.

Initial conditions
    x[0] = x0
    y[0] = y0
$$x(0) = x_0$$ $$y(0) = y_0$$

Drop the starting populations into the first slot of each array. Note x[0], not x[1] — Python is zero-indexed, unlike MATLAB. We don't need to set t[0] because np.zeros already put a zero there, and time starts at 0.

The integration loop — the heart of Euler's method
    for n in range(n_steps):
        x[n+1] = x[n] + dt * dxdt(x[n], y[n])
        y[n+1] = y[n] + dt * dydt(x[n], y[n])
        t[n+1] = t[n] + dt
Definition of the derivative: $$\frac{dx}{dt} \approx \frac{x_{n+1} - x_n}{\Delta t}$$ ↓ rearrange for $x_{n+1}$ $$x_{n+1} = x_n + \Delta t \cdot \frac{dx}{dt}$$ ↓ substitute the Lotka–Volterra RHS $$x_{n+1} = x_n + \Delta t (\alpha x_n - \beta x_n y_n)$$ $$y_{n+1} = y_n + \Delta t (\delta x_n y_n - \gamma y_n)$$ $$t_{n+1} = t_n + \Delta t$$

The first two code lines are literal translations of the final equations. The whole derivation comes from one idea: rearrange the definition of the derivative to solve for the next state. This single move is the heart of every numerical ODE solver in this document — RK4 in the next section builds on the exact same starting point, just with a smarter choice of which slope to use on the right-hand side.

Important: both updates use x[n] and y[n] — not the freshly computed x[n+1]. The two equations are coupled and must use the same "before" state, otherwise the predator update would be reading a half-stepped prey value. The third line advances the clock; that part is bookkeeping, not integration.

Return the result
    return t, x, y

Hand back all three arrays as a tuple. The caller unpacks them in one line: t, x, y = euler_method(...). After the call, you have everything needed to plot populations versus time or as a phase portrait.

4

4th-order Runge–Kutta

▾

Euler trusts the slope at the start of the interval for the entire step. RK4 doesn't. It samples the slope four times — once at the start, twice at the midpoint (each refining the previous), and once at the end — then takes a weighted average. The four sample points are chained: each one uses the previous slope to estimate where to look next.

The math below uses $f_x$ and $f_y$ as shorthand for the right-hand sides of the Lotka–Volterra equations: $f_x(x, y) = \alpha x - \beta x y$ and $f_y(x, y) = \delta x y - \gamma y$. In the code, these are dxdt and dydt.

Same setup as Euler
def rk4_method(x0, y0, dt, t_final):
    n_steps = int(t_final / dt)
    t = np.zeros(n_steps + 1)
    x = np.zeros(n_steps + 1)
    y = np.zeros(n_steps + 1)
    x[0] = x0
    y[0] = y0

Identical to euler_method: compute the number of steps, pre-allocate arrays for time, prey, and predator, and write the initial conditions into slot 0. The structure of every fixed-step ODE solver is the same — only the update rule inside the loop changes.

Open the loop and snapshot the current state
    for n in range(n_steps):
        xn, yn = x[n], y[n]

Each pass advances from index n to n+1. We copy the current state into the local names xn and yn to avoid writing x[n] and y[n] over and over in the four stages that follow — both clearer to read and slightly faster.

Stage 1 — slope at the start of the interval
        k1x = dxdt(xn, yn)
        k1y = dydt(xn, yn)
$$k_{1x} = f_x(x_n, y_n)$$ $$k_{1y} = f_y(x_n, y_n)$$ evaluated at: $t_n$, state $(x_n, y_n)$

The slope right where we are. This is the same thing Euler uses — and if we stopped here and updated with $x_{n+1} = x_n + \Delta t \cdot k_1$, we'd be Euler. RK4's improvement comes from not stopping here.

Stage 2 — slope at the midpoint, estimated using k₁
        k2x = dxdt(xn + 0.5*dt*k1x,
                   yn + 0.5*dt*k1y)
        k2y = dydt(xn + 0.5*dt*k1x,
                   yn + 0.5*dt*k1y)
$$k_{2x} = f_x(x_n + \tfrac{1}{2}\Delta t \cdot k_{1x},\ y_n + \tfrac{1}{2}\Delta t \cdot k_{1y})$$ $$k_{2y} = f_y(x_n + \tfrac{1}{2}\Delta t \cdot k_{1x},\ y_n + \tfrac{1}{2}\Delta t \cdot k_{1y})$$ evaluated at: $t_n + \tfrac{1}{2}\Delta t$, state estimated by an Euler half-step using $k_1$

To find the slope at the midpoint, we first need to know where the system is at the midpoint. We don't know exactly, but we can estimate it: take a half-step forward using the slope we already have ($k_1$), then evaluate the slope at that estimated midpoint state.

This is also RK2 by itself. If you stopped here and updated with $x_{n+1} = x_n + \Delta t \cdot k_2$, you'd have the midpoint method — already much better than Euler. RK4 keeps going.

Stage 3 — slope at the midpoint again, refined using k₂
        k3x = dxdt(xn + 0.5*dt*k2x,
                   yn + 0.5*dt*k2y)
        k3y = dydt(xn + 0.5*dt*k2x,
                   yn + 0.5*dt*k2y)
$$k_{3x} = f_x(x_n + \tfrac{1}{2}\Delta t \cdot k_{2x},\ y_n + \tfrac{1}{2}\Delta t \cdot k_{2y})$$ $$k_{3y} = f_y(x_n + \tfrac{1}{2}\Delta t \cdot k_{2x},\ y_n + \tfrac{1}{2}\Delta t \cdot k_{2y})$$ evaluated at: $t_n + \tfrac{1}{2}\Delta t$, state estimated by a half-step using $k_2$

Same time point as stage 2 — the midpoint — but we use the better slope estimate ($k_2$) to take the half-step that gets us there. So $k_3$ is a refined version of $k_2$: a second, sharper guess at what the midpoint slope really is.

This is what makes RK4 fourth-order rather than second-order: refining the midpoint estimate with itself cancels out a whole layer of error.

Stage 4 — slope at the end of the interval, using k₃
        k4x = dxdt(xn + dt*k3x,
                   yn + dt*k3y)
        k4y = dydt(xn + dt*k3x,
                   yn + dt*k3y)
$$k_{4x} = f_x(x_n + \Delta t \cdot k_{3x},\ y_n + \Delta t \cdot k_{3y})$$ $$k_{4y} = f_y(x_n + \Delta t \cdot k_{3x},\ y_n + \Delta t \cdot k_{3y})$$ evaluated at: $t_n + \Delta t$, state estimated by a full step using $k_3$

Now we use our best midpoint slope ($k_3$) to take a full step, and measure the slope at the end of the interval. Note this is dt, not 0.5*dt — $k_4$ samples the endpoint, not the midpoint.

We now have four slope estimates: one at the start, two refined estimates at the middle, and one at the end. Time to combine them.

The weighted-average update — Simpson's rule in disguise
        x[n+1] = xn + (dt/6.0) * (k1x + 2*k2x
                                + 2*k3x + k4x)
        y[n+1] = yn + (dt/6.0) * (k1y + 2*k2y
                                + 2*k3y + k4y)
        t[n+1] = t[n] + dt
$$x_{n+1} = x_n + \frac{\Delta t}{6}(k_{1x} + 2k_{2x} + 2k_{3x} + k_{4x})$$ $$y_{n+1} = y_n + \frac{\Delta t}{6}(k_{1y} + 2k_{2y} + 2k_{3y} + k_{4y})$$ $$t_{n+1} = t_n + \Delta t$$

Each $k$ is a slope. The weighted average $(k_1 + 2k_2 + 2k_3 + k_4)/6$ is the "best" single slope to use for the whole interval — and multiplying by $\Delta t$ converts that slope into a position change. The structure is the same as Euler (position += step × slope), just with a much better slope. The clock advance on the last line is bookkeeping, not part of the integration scheme.

Why these specific weights? Compare with Simpson's rule from numerical integration:

Simpson: $$\int_a^b f\,dx \approx \frac{h}{6}(f_{\text{start}} + 4 f_{\text{mid}} + f_{\text{end}})$$ RK4: $$\Delta x \approx \frac{\Delta t}{6}(k_1 + 2k_2 + 2k_3 + k_4)$$

Same pattern. The "$4 f_{\text{mid}}$" of Simpson's rule splits into "$2k_2 + 2k_3$" because RK4 has two midpoint estimates instead of one. The factor of $1/6$ is identical. This is not a coincidence — RK4 can be derived as Simpson's rule applied to the integral form of the ODE.

Return the result
    return t, x, y

After the loop finishes, hand back all three arrays as a tuple. The caller unpacks them in one line: t, x, y = rk4_method(...). Total cost per step: four function evaluations instead of Euler's one, in exchange for accuracy that scales as $\Delta t^5$ instead of $\Delta t^2$. For any problem where accuracy matters, that trade is overwhelmingly worth it.

5

Comparison: Euler vs RK4

▾

Two solvers, same code skeleton, different update rules — and meaningfully different behavior. This section runs both and compares them visually. The goal isn't just to declare a winner; it's to see how Euler fails so that you recognize the failure mode in your own work later.

Each figure in this section is preceded by the actual matplotlib code that produced it, annotated the same way as the solvers. By the end, you'll have seen every plotting pattern you need to reproduce the experiments yourself — from a basic single-panel figure to a 2×2 grid with subplots and phase portraits.

First impression — Euler looks fine

Run Euler with $x_0 = y_0 = 1$, $\Delta t = 0.01$, and a 50-time-unit horizon. Plot prey and predator versus time, with dashed reference lines at the equilibrium values $\gamma/\delta = 2.5$ and $\alpha/\beta = 2.5$. The annotated code below is exactly what produced the figure underneath it — copy it, run it, and you should see the same thing.

Call the solver
x0, y0 = 1.0, 1.0
dt = 0.01
t_final = 50

t, x, y = euler_method(x0, y0, dt, t_final)

Set the initial populations, step size, and final time, then call the solver. Starting at $(1.0, 1.0)$ places us off the equilibrium $(2.5, 2.5)$, which is necessary if we want to see oscillations — at the equilibrium, nothing moves.

The notation x0, y0 = 1.0, 1.0 is Python's tuple unpacking: the right-hand side builds a tuple and the left-hand side splits it into two variables. The t, x, y = euler_method(...) line uses the same trick to unpack the three arrays the solver returns — one line, three return values.

Create the figure and plot the two populations
fig, ax = plt.subplots()

ax.plot(t, x, label='Prey (x)',
        color='#185FA5', linewidth=1.5)
ax.plot(t, y, label='Predator (y)',
        color='#A32D2D', linewidth=1.5)

plt.subplots() creates a new figure with one set of axes inside it, returning two objects: fig (the whole figure, like a canvas) and ax (the plot area where the data goes). For a simple single-panel plot you only need ax; fig matters when you want to save the figure or arrange multiple panels.

Each ax.plot call draws one line. The first two arguments are the x-values and y-values; everything after that is styling. label sets the name shown in the legend, color takes any CSS-style hex code, and linewidth is in points. Both calls plot against t on the horizontal axis.

This is the modern matplotlib pattern. Older code uses plt.plot(...) directly, which works but doesn't scale well to multi-panel figures. The explicit fig, ax style is worth learning first.

Add equilibrium reference lines and finish the figure
ax.axhline(gamma/delta, color='#185FA5',
           linestyle='--', alpha=0.4,
           label='Prey equilibrium')
ax.axhline(alpha/beta, color='#A32D2D',
           linestyle='--', alpha=0.4,
           label='Predator equilibrium')

ax.set_xlabel('Time')
ax.set_ylabel('Population')
ax.set_title("Lotka–Volterra: Euler's Method (dt = 0.01)")
ax.legend()

$$x^* = \gamma/\delta = 2.5$$ $$y^* = \alpha/\beta = 2.5$$

axhline draws a horizontal line across the entire plot at a given y-value. We use it to mark the equilibrium populations as visual references — the oscillations should be centered on these lines. linestyle='--' makes them dashed and alpha=0.4 makes them partly transparent so they sit behind the data.

The finishing touches: set_xlabel and set_ylabel name the axes, set_title puts a title on top, and legend() draws the legend using whatever label= values were passed earlier — that's why we set them on each plot and axhline call. grid(True, alpha=0.3) turns on a faint background grid. plt.show() displays the figure; in a Jupyter notebook it's technically optional, but including it makes the script work the same way as a standalone .py file.

This is exactly what the equations predict qualitatively. If we stopped here we'd conclude Euler works. But qualitative agreement is a low bar — we should run the simulation longer and look at it from a different angle before believing it.

Run it longer, plot the phase portrait — Euler spirals

Now run Euler with a smaller step ($\Delta t = 0.001$) for twice as long (100 time units), and plot the phase portrait on the right — predator on the y-axis, prey on the x-axis. For a system like this, the true solution traces out a closed loop in the phase plane: each oscillation should bring you back to where you started.

Re-run Euler with a smaller step and a longer horizon

Same solver, different settings. The variables are renamed (t_big, x_big, y_big) so this run doesn't overwrite the previous one — we'll need them both if we want to compare later.

The title uses an f-string with {dt_big} to interpolate the actual step size into the title text. This is a small habit worth picking up: titles that embed the settings they represent make it much harder to confuse two similar-looking plots later.

Right subplot — the phase portrait

axes[1].set_xlabel('Prey (x)')

The phase portrait trick. A phase portrait plots the state variables against each other instead of against time. Look at the first plot call: axes[1].plot(x_big, y_big) — prey on the x-axis, predator on the y-axis, no time axis at all. The same plot function does both; only the arguments change.

The second plot call draws a single point at the equilibrium. Passing 'o' as the third positional argument tells matplotlib to draw markers instead of a line, and markersize controls how big they are. This is a shorthand format string — you can also pass things like 'r--' (red dashed line) or 'gs' (green squares).

plt.tight_layout() automatically adjusts the spacing between subplots so titles and labels don't overlap. Cheap insurance — always call it before plt.show() on multi-panel figures.

What you're seeing

This is not a bug. The code is correct. What you're seeing is a fundamental property of Euler's method: at every step, it extrapolates a straight line from the current slope, but the true trajectory is curving. That tiny mismatch — the gap between a straight line and a curve — accumulates. For the Lotka–Volterra system specifically, the accumulated error always pushes outward, away from the equilibrium, so the phase portrait spirals rather than closing.

Making $\Delta t$ smaller helps, but it does not fix the underlying problem. You would need an extraordinarily small time step — and a correspondingly long runtime — to hide the drift. The right fix is not a smaller step, but a smarter step. That's what RK4 is for.

Head-to-head — Euler vs RK4

Run both methods with identical settings: same initial conditions, same parameters, same time step ($\Delta t = 0.0001$), same final time ($t_{\text{final}} = 100$). Compare them side by side in a 2×2 grid — populations vs time on top, phase portraits on the bottom.

Run both solvers
x0, y0 = 1.0, 1.0
dt = 0.0001
t_final = 100

t_e, x_e, y_e = euler_method(x0, y0, dt, t_final)
t_r, x_r, y_r = rk4_method(x0, y0, dt, t_final)

Two calls, identical arguments. The naming convention _e for Euler and _r for RK4 keeps the two runs' arrays distinct in the workspace — important because we're about to plot them side by side and any mixup would make the comparison meaningless.

The step size is deliberately small ($\Delta t = 0.0001$) and the horizon long ($t_{\text{final}} = 100$). We want to give Euler every chance to look good and still see it fail, because the point is to show that even at tiny step sizes the drift doesn't go away — it's structural.

Create a 2×2 subplot grid
fig, axes = plt.subplots(2, 2, figsize=(14, 10))

This is the same indexing pattern as a NumPy 2D array or a MATLAB matrix: first index is the row, second is the column. If you come from MATLAB, the only adjustment is the zero-based counting.

Notice that the equilibrium marker is drawn identically on both phase portraits. That's intentional: it gives you a fixed visual reference. On a correct integration the trajectory should close around it; on Euler the trajectory pulls away from it.

Quantifying the difference

Looking at plots is useful, but numbers are conclusive. Using scipy.signal.find_peaks to locate each oscillation's peak in the prey population, we can measure whether successive peaks are growing (Euler drifting) or staying constant (RK4 conserving):

MethodPeaks foundFirst peakLast peakDrift
Euler107.67767.6926+0.19%
RK4107.67707.67700.0000%

Over 100 time units and roughly 10 oscillations, Euler's prey peaks have grown by about 0.19 percent — small, but measurably and consistently increasing. RK4's peaks are identical to four decimal places from first to last. And remember: this is at $\Delta t = 0.0001$, already a very small step. At larger steps the Euler drift becomes dramatic; at smaller steps it persists but slows. The difference is structural, not a matter of resolution.

The takeaway

Both solvers share the same skeleton. The only thing that changed between them is the update rule inside the loop — everything else, from the array allocation to the tuple return, is identical. That's the most important pattern in this whole document: every fixed-step ODE solver you will ever write or read is a variation on this template, and the part that varies is always the choice of which slope to use. Master that one idea and the rest is mechanical.
